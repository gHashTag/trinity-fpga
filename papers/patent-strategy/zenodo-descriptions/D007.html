Problem Statement

Matrix-vector multiplication with ternary {-1, 0, +1} weights should require zero multiplications — each weight contributes addition, subtraction, or nothing. Yet standard implementations use branching (if/else per weight) that causes pipeline stalls and poor vectorization. Existing quantization methods (BitNet b1.58/Ma 2024, GPTQ/Frantar 2023, AWQ/Lin 2024) target GPU inference with CUDA kernels and do not provide CPU-optimized branchless or SIMD-vectorized variants for ternary weights.

Technical Disclosure

Four Matrix-Vector Multiply Variants

Variant 1: Packed 2-bit Encoding

Each ternary weight encoded as 2 bits: +1 → 01, 0 → 00, −1 → 11. Pack 16 weights into one u32. Decode via mask-and-shift: sign = (packed >> (2*i)) & 3. Memory: 1.95M params × 2 bits = 488 KB (vs 1.95 MB at i8 = 4× compression).

Variant 2: Branchless Bit-Manipulation (9.2× speedup)

Eliminate all branches using arithmetic identity:

// For weight w ∈ {-1, 0, +1} encoded as 2 bits (sign, nonzero):
result += (input ^ sign_mask) + (sign_mask & 1)  // conditional negate
result &= nonzero_mask                            // zero if w=0

Zero branches → zero pipeline stalls → 9.2× measured speedup over naive branching implementation on x86_64.

Variant 3: Sparse CSR Format

Compressed Sparse Row stores only non-zero weights with column indices and sign bits. Effective when zero-density > 50%. Skips zero weights entirely — useful for pruned ternary models.

Variant 4: SIMD Vectorized

Two sub-variants:

  • f16 SIMD: 16-wide vector operations, process 16 accumulations per instruction
  • f32 SIMD: 8-wide vector operations, higher precision

Combined speedup: 4–33× depending on dimension and sparsity.

Obvious Extensions (Picket Fence)

  • GPU CUDA kernels: porting branchless variant to CUDA with warp-level primitives
  • ARM NEON: NEON-specific SIMD intrinsics for mobile/edge deployment
  • WebAssembly SIMD: browser-based ternary inference using WASM SIMD128
  • Mixed-precision ternary+int4: hybrid approach where attention uses ternary, FFN uses int4
  • Batch matrix multiply: extending single-vector variants to batch processing
  • Auto-tuning: runtime selection of optimal variant based on matrix shape and hardware
  • Ternary sparse attention: applying CSR format to attention weight matrices
  • AVX-512 variant: wider SIMD for server-class CPUs (32 f16 or 16 f32 per instruction)
